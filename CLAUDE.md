### 委派工作流（2026-06-12 用户确认，验证有效）

两种排法按任务类型分流：
- **判断密集型**（统计建模、含诚实性判断）：**Fable 建 + Opus 审**（Opus 冷启动新视角，每轮都抓出主脑盲区）
- **规格明确、实现量大**（聚合/展示/工程，如 benchmark 记分卡）：**Fable 详细计划 → Opus 建 → Fable 审**（卸主脑的载）

六步协议（关键是新增"审计划"）：
```
① 主脑写计划 → ② 独立审计划(挑 spec 洞，比建完再改便宜得多) → ③ 建造
→ ④ 独立审实现 → ⑤ 修复 → ⑥ 验证提交
```
- **审实现按风险分级**：公开统计结论/模型链=双审(Fable+全新Opus)；常规聚合/展示=单审即可，别无脑双审
- **铁律：审查者 ≠ 建造者**（不同模型/全新上下文，保证独立）
- 实证：benchmark 那次 Opus 代码很干净，但 Fable 审出了 Opus 自测漏掉的 ordering bug；而那个 bug 本可在"审计划"阶段更早抓到——所以两道审都有价值
