## P1 — 模型方法论（核心价值所在，约 1-2 周）

诚实追踪已经显示 Tier≥4 样本外平均 -1.3pp——模型目前没有被证实的优势。
以下按"对结论影响大小"排序。

- [ ] **1. 先把"概率"定义清楚（最重要）**
  现状混用了三个不同事件的频率：`MONTHLY_PRIOR` 是"当月整月上涨"的概率（月频事件），
  `DOW_LR` 是"当天上涨"的胜率比（日频事件），walk-forward 验证的是"未来 20 日上涨"（20 日事件）。
  三者在 log-odds 里连乘，得到的数字没有明确定义的事件。
  方案：统一目标为 **P(未来20日上涨)**——先验改用"从该月某日起 20 日窗口上涨频率"，
  所有 LR 都以同一事件估计（walk_forward.py 的 `learn_lrs` 已经是这么算的，把它变成唯一来源）。
  前端文案同步说明"20日窗口"，否则 58% 会被读成"明天涨的概率"。

- [ ] **2. 概率比 ≠ 贝叶斯因子**
  `bayesian_update()` 在 log-odds 空间累加 `log(lr)`，但所有 LR 都是 **胜率比** `wr_on/base`
  （如 52.4%→49.3% 得 0.94），不是胜算比。正确的 odds-ratio 应为
  `(wr_on/(1-wr_on)) / (base/(1-base))`（同例应为 0.88）。当前做法系统性低估每个因子的更新幅度，
  且偏差随因子强度非线性放大。
  方案 A（小改）：换成 odds-ratio。
  方案 B（推荐，一步到位）：放弃手搓朴素贝叶斯，用 **L2 正则逻辑回归** 对全部二值因子建模
  （sklearn 已在依赖里），系数即对数胜算比，天然处理因子相关性（见下条），walk-forward 直接验证它。

- [ ] **3. 因子相关性重复计数**
  朴素贝叶斯独立性假设在这里明显不成立：RSI 超卖、MA200 下方、高波动、VIX 倒挂高度相关
  （恐慌时同时触发），DOW×WOM×假日×税季在同一段样本上重叠估计。相关因子连乘 = 同一份证据算多次，
  概率被推向极端。逻辑回归（上条方案 B）顺带解决；若保留朴素贝叶斯，至少对相关因子分组取最强者。

- [ ] **4. 只展示校准后概率，档位也用校准值定**
  现状：原始 58.3% 显示为"第3档·中性"，旁边又写"校准后65%"——65% 落在第 4 档区间，
  同一天两个数字给出两个相反的操作含义。原始输出应视为内部得分，
  对外只展示校准值 + 明确时间范围（"未来20日上涨概率约65%"），`_tier()` 吃校准后的值。
  注意：校准曲线目前是样本内的，应改为 walk-forward 各折测试期数据拼出的样本外校准。

- [ ] **5. 显著性检验换块自助法（block bootstrap）**
  重叠 20 日窗口让 t 检验 p 值乐观约一个数量级（代码注释已诚实承认）。
  用 stationary/circular block bootstrap（块长 ~20）重算 backtest 与 walk-forward 的置信区间，
  顺带对日历异象做多重检验校正（Benjamini-Hochberg 即可）——税日 66%、九月效应这类
  都是文献里被数据挖掘批评过的经典对象。

- [ ] **6. 按样本外证据修剪因子**
  walk-forward 已输出每因子 LR 与 n。给每个因子算"各折测试期是否方向一致"，
  不一致的（大概率包括主观事件 LR：halving 1.15、gold_spike 0.88 等）从生产模型移除或归零。
  主观 LR 若保留，前端必须标注"主观估计，未经验证"（目前混在一起显示）。

- [ ] **7. 未来窗口预测的技术因子冻结问题**
  `find_next_opportunities` 把当天的 RSI/MA/波动率冻结后外推 130 个交易日，
  但面板文案写"未来45个交易日"、日历标题写"未来60天"。三个数字对齐为一个（建议 ~40 个交易日，
  技术因子冻结的合理寿命），超出部分只用日历因子并在 UI 上注明"仅日历先验"。

## P2 — 前端信息架构与体验（约 1 周）

单页堆了左栏 14 个面板 + 右栏 16 个图表区，没有导航。"今天该不该买"这一个问题的答案散落在
信号环、简报、双指数、卖出信号、机会窗口五个地方。

- [ ] **信息分组 + 锚点导航**
  顶部加 sticky 导航，把内容分四组：
  ① 今日决策（信号环/简报/双指数/卖出/未来窗口）
  ② 验证与问责（实盘追踪/模拟盘/回测/walk-forward/模型体检）
  ③ 研究面板（隔夜分解/月度胜率/假日/熊市/多元分析/日历规律）
  ④ 个人工具（持仓计算器/CGT/SpaceX/我的模拟盘）
  个人工具与公开研究站性质不同，建议独立成 `tools.html`（站点是公开的 GitHub Pages）。

- [ ] **文案与数据一致性清理**
  - 模拟盘面板标题"四策略竞技"，实际五个策略（buyhold/trend/signal/momentum/overnight），
    `paper_trading.py` 里 note 也写"四个策略"——三处统一
  - 档位阈值在 `build_signals._tier()`、`app.js TIER_META`、index.html 档位图例三处各写一份，
    改一处必漏两处：由流水线把阈值写进 signals.json，前端读取渲染
  - "未来60天/45个交易日/130天"对齐（见 P1-7）

- [ ] **样式收敛**
  index.html 几乎每个 div 都带 inline style，style.css 只有 208 行。把重复的 inline 样式
  （panel 内边距、小字号、分隔线）收进 class。顺带统一字号体系：现在 0.65-0.9rem 共出现十几档，
  收敛到 4 档（标题/正文/辅助/脚注），0.72rem 以下的正文阅读性太差。

- [ ] **性能**
  Plotly 完整版 CDN ~3.5MB，是首屏最大成本。换 `plotly.js-basic-dist-min`（散点/柱状/热力图够用，~1MB）；
  首屏外图表用 IntersectionObserver 懒渲染（标签页已做了懒加载，把同样机制推广到滚动方向）。

- [ ] **可访问性最低线**
  标签页是 div+onclick：换 `<button role="tab">` 并支持键盘；档位/涨跌不能只靠颜色区分
  （加 ▲▼ 或文字，红绿色盲约占男性 8%）；交互控件补 aria-label。

## 不建议做的（明确排除，防止范围蔓延）

- **不要加更多因子**。现有 11 个因子样本外合计 -1.3pp，问题是方法论不是因子数量。
- **不要上深度学习**。日频 ~6,500 样本、信噪比极低，过拟合是必然结局；`--full` 里的
  XGBoost/Prophet 留作研究面板即可，不要进信号链路。
- **不要引入前端框架**。无构建的静态站是这个项目的部署优势，拆模块就够了。
